# Swarm-Scale vNext — Verification Matrix

> Definition of done for the swarm-scale operator control plane (epic
> `bd-2mb03`). Every implementation bead in this program must pass the
> gates in this document before it can be marked closed; downstream
> agents reading this file should be able to run the matrix without
> reading the original idea-wizard discussion.

## Why this exists

Implementation beads in `bd-2mb03` add concurrent, observability, and
gating behavior that is hard to verify with unit tests alone. Without a
shared matrix, agents tend to rely on proxy signals (build green,
package compiles) and miss regressions in latency, memory, contract
shape, and replay fidelity. This matrix anchors verification to
concrete commands, artifacts, and triage notes so each gate has the
same semantics across agents.

## Gate taxonomy

Every gate falls into one of four categories. Each implementation bead
declares which gates apply to it (see the per-bead checklist below).

| Gate         | Purpose                                              | Default invocation |
|--------------|------------------------------------------------------|--------------------|
| `short`      | Deterministic correctness; runs in `-short` mode.    | `go test -short ./...` |
| `race`       | Concurrency correctness on touched packages.         | `go test -race ./internal/<pkg>/...` |
| `bench`      | Lower-bound throughput / latency check.              | `go test -run=^$ -bench=. -benchmem ./internal/<pkg>/...` |
| `load`       | Opt-in 100+ pane synthetic swarm.                    | `NTM_SWARM_LOAD=1 go test -run TestLoad -timeout 10m ./...` |

`short` runs locally; `race` and `bench` always go through `rch exec --
go test -race ...` to avoid local CPU contention; `load` is opt-in and
must run on a host with the configured baseline (>= 32 cores, >= 64GB
RAM) to produce comparable artifacts.

## Logging requirements

Every gate writes structured log lines (slog) with these correlation
fields (omitted only when not applicable):

- `test_run_id` — UUIDv7 generated by the harness once per matrix run.
- `gate` — one of `short|race|bench|load`.
- `feature` — the bead-id of the feature under test (e.g. `bd-2mb03.1`).
- `session` — synthetic swarm session name when applicable.
- `pane_count` — total tmux/synthetic panes active.
- `command_count` — robot commands issued during the gate.
- `event_count` — Agent Mail / pipeline events handled.
- `pressure_state` — overall pressure level (`low|normal|elevated|high|critical`).

Tests that omit any of `test_run_id`, `gate`, `feature` are out of
matrix compliance and must be patched before close.

## Artifact formats

All artifacts land under `artifacts/<test_run_id>/<bead-id>/<gate>/` so
matrix runs can be archived and compared.

| Signal              | File              | Format / fields |
|---------------------|-------------------|----------------|
| Latency percentiles | `latency.json`    | `{"p50_ms": float, "p95_ms": float, "p99_ms": float, "max_ms": float, "samples": int}` |
| Memory growth       | `mem.json`        | `{"baseline_mb": float, "peak_mb": float, "delta_mb": float}` |
| Goroutines          | `goroutines.json` | `{"baseline": int, "peak": int, "leaked": int}` |
| Robot contract      | `contract.json`   | `{"surface": string, "failures": [{"command": string, "field": string, "want": any, "got": any}]}` |
| Timeline replay     | `timeline.jsonl`  | One event per line: `{"ts": rfc3339, "kind": string, "session": string, "pane": int, "thread_id": string, "bead_id": string}` |
| Queue-dry diag      | `queuedry.json`   | `{"empty": bool, "stale_in_progress": int, "active_reservations": int, "sync_healthy": bool, "recommendations": [string]}` |

Failure triage convention: every failing gate must produce
`failure.txt` with the gate command, exit code, last 200 lines of
stderr, and the artifact paths consulted. Triage scripts read this
file first.

## Run sequencing

A full matrix run executes gates in this order to fail fast and cache
intermediate state:

1. `short` for every bead in scope.
2. `race` for beads marked Required.
3. `bench` for any opt-in beads (parallel, rch).
4. `load` once per matrix run, with `NTM_SWARM_LOAD=1`.

If `short` fails for any bead, the matrix stops and writes
`failure.txt`; the operator should triage with `cat artifacts/<run>/<bead>/short/failure.txt`.

## Required vs no-real-model paths

- **No real model:** every `short` and `race` gate runs entirely with
  the synthetic harness; no real Claude / Codex / Antigravity / Gemini CLI is
  launched. This is mandatory for matrix reproducibility.
- **Opt-in 100+ pane:** `load` gate spawns 200 synthetic panes via
  `bd-2mb03.2`. Triggered only when `NTM_SWARM_LOAD=1` is exported.

A future agent who wants only the no-real-model gates can run:

```bash
go test -short ./internal/{pressure,profiler,coordinator,robot/contract}/...
```

## Surfacing this matrix in `bv --robot-triage`

Adding this matrix bead must not hide implementation work. Verify with:

```bash
bv --robot-triage | jq '.recommendations | length'
```

The count must remain >= the count before this matrix bead landed; if
it drops, an implementation bead inadvertently became dependent on
this verification bead and the dependency must be reversed.

## Change control

Edits to this matrix follow the same workflow as code:

1. Open a bead under `bd-2mb03` referencing the change reason.
2. Update this file.
3. Run all `short` gates listed above to confirm nothing relies on the
   old wording.
4. `br sync --flush-only && git add docs/verification-matrix-swarm-scale-vnext.md .beads/issues.jsonl && git commit`.

This document supersedes any inline test-strategy comments inside
swarm-scale beads.
